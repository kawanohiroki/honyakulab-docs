# AIがドキュメントを読む時代へ——GitBookが提示した「フィードバックループを閉じる」戦略

***

**出典** セミナータイトル：Closing the Docs Feedback Loop&#x20;

視聴ページ：https://www.gitbook.com/events/closing-the-docs-feedback-loop

***

### はじめに

2026年4月7日、ドキュメント管理プラットフォームのGitBookがオンラインウェビナー「Closing the Docs Feedback Loop（ドキュメントのフィードバックループを閉じる）」を開催しました。Developer RelationsのAddison氏が登壇し、新機能のライブデモとQ\&Aを合わせた約1時間のセッションです。

今回のリリースは単なる機能アップデートにとどまりません。ドキュメントの役割が根本から変わりつつあるという認識に基づき、それに対応するための「構造的な仕組み」を提供するものでした。本記事では、セミナーの内容を論点ごとに詳細にまとめます。

***

### 1. 背景：AIがドキュメントの主要な「読者」になった

GitBookは毎年「State of Documentation Report（ステート・オブ・ドキュメンテーション・レポート）」を公開しています。2026年版では、特に注目すべき数字が示されました。

**GitBook上のドキュメントサイトにおけるAI閲覧比率が40%超に達した**というものです。

これが意味することは非常に大きいです。ChatGPT、Claude、Cursor、その他のAIエージェントが、ドキュメントサイトにアクセスして情報を参照するケースが急増しており、もはや「人間だけがドキュメントを読む」という前提は崩れています。

テクニカルライターや文書管理者が書くコンテンツの影響範囲は、AIを経由することで従来の何倍にも拡大しています。ドキュメントに誤りがあれば、そのAIが誤情報を広め、無数のエンドユーザーに伝わってしまいます。逆に、ドキュメントが充実していれば、AIを通じた価値提供が飛躍的に広がります。

#### ドキュメントが陳腐化する問題はより深刻に

ドキュメントが古くなるという問題は以前から存在しましたが、AI活用が広がった現在では重要度がさらに増しています。製品開発チームはかつてないスピードでフィーチャーをリリースしており、ドキュメントが実態に追いつかない状況が日常的に起きています。GitBook自身もその課題に直面していると、Addison氏は率直に述べています。

***

### 2. 全体コンセプト：フィードバックループを閉じる

今回のリリースの核心は「Closing the Docs Feedback Loop」というコンセプトです。これはフライホイール（自己強化ループ）として説明されました。

1. **ドキュメントの質が高い** → AIアシスタントの回答精度が上がる
2. **回答精度が上がる** → ユーザーが積極的に質問するようになる（質問数が増加）
3. **質問データが蓄積される** → ドキュメントのギャップ（何が書かれていないか）が可視化される
4. **ギャップを埋める** → さらにドキュメントの質が向上する
5. → 1に戻る

このループを回すために今回リリースされた機能が、**AI Insights（AIインサイト）**、**Connections（コネクション）**、**Embeddable Assistant（埋め込みアシスタント）**、\*\*Channels（チャンネル）\*\*の4つです。それぞれをデモを交えながら詳しく説明します。

***

### 3. AI Insights（AIインサイト）

#### 概要

AI Insightsは、GitBook Assistantに対してユーザーが行った質問を収集・分析し、ドキュメントの回答カバレッジをスコア化する新機能です。

以前から「Analytics（アナリティクス）」パネルがありました。これはページビュー、検索クエリ、壊れたリンクなどを表示するものでしたが（名称はそのまま残ります）、今回あらたに「Insights（インサイト）」パネルが追加され、AI利用に関する情報が集約されるようになりました。

#### アクセス方法

GitBookエディタの左側パネルから対象のドキュメントサイトを選択し、上部の「Insights」タブを開くことで確認できます。

#### ダッシュボードで確認できること

ダッシュボードのトップには以下の情報がサマリー表示されます。

**Estimated Savings（推定削減時間）** GitBook Assistantが質問に正常回答した件数に基づき、サポートチームが節約できた時間の推計スコアです。Addison氏はこの算出方法についての詳細なカスタマイズは現時点では不可と述べつつも、サポートコスト削減の可視化として有用だと説明しました。

**質問数（総数・回答済み件数）** アシスタントに投げられた総質問数と、実際に回答できた質問数がそれぞれ表示されます。「何件聞かれて、何件答えられたか」が一目でわかります。

#### トピック別自動分類

GitBookは質問内容をAIが自動でトピックに分類し、ランク付けします。デモでは「Syncly」という架空のカレンダーアプリのドキュメントを使っており、以下のようなトピックが自動生成されていました。

* Enterprise関連の質問
* カレンダー関連の質問
* サポート関連の質問
* 通知（Notifications）
* リリースノート
* ローカリゼーション

これらは質問内容から自動的に決まるため、サイトごとにトピックは異なります。

各トピックには以下の情報も紐付けられます。

**質問タイプの自動ラベル**

* Troubleshooting（問題解決）
* Exploring（製品発見・リード獲得目的の探索）
* How-to（使い方の確認）

の3種に自動分類されます。

**回答状況のフィルター**

* Answered（回答済み）
* Partially Answered（一部回答）
* Unanswered（未回答）

で絞り込むことができます。

#### トピック詳細ビュー

トピックをクリックすると、AI生成によるトピックのサマリー、配下の個別質問一覧と質問頻度、各質問への回答評価（ユーザーからの「役立つ」フィードバックの反映）が確認できます。

#### 個別質問詳細ビュー

さらに個別の質問をクリックすると、きわめて詳細な情報が確認できます。

* 質問が投げられた日時
* 自動分類カテゴリ
* 回答パフォーマンス（Answered / Not Answered）
* 質問の発生元（ドキュメントサイト上か、埋め込みアシスタントか等）とその比率
* 使用されたコネクション数
* 訪問者のプロフィール（国・言語）
* アシスタントが実際に行った会話ログ全文
* 回答に使用されたドキュメントページの一覧（チェンジログ、コミュニティ、クイックスタート等）

デモでは「How do I create a calendar invite?（カレンダーの招待状はどうやって作るの？）」という質問が「Answered（回答済み）」と評価されており、回答に使われたページが複数表示されていました。

#### 実践的な活用方法

「Unanswered」フィルターを使うと、未回答質問の一覧が得られます。デモでは「Is there a trial available?（トライアル版はありますか？）」という質問が繰り返し未回答になっていることが判明。ドキュメント内にトライアルに関する情報が存在しないためです。

こうした未回答質問の蓄積こそが、「次にどのページを書くべきか」の優先順位付けに使える具体的なシグナルになります。また、各トピックのコンテンツ量（ページ数・レコード数）と回答率を対比することで、「コンテンツが少ないのに未回答が多い」分野を特定し、集中的に強化する判断が可能になります。

なお、AI Insightsパネルに表示される情報はリアルタイムで更新されます。デモでは、その場で参加者にアシスタントを使ってもらい、質問がリアルタイムにインサイトパネルに反映される様子が実演されました。

### 4. Connections（コネクション）

#### 概要

Connectionsは、GitBook Assistantが参照できる情報源を、公式ドキュメント以外にも拡張する機能です。

ドキュメントサイトにすべての情報がまとまっていれば理想的ですが、現実には情報はさまざまな場所に分散しています。ブログ記事、YouTubeの解説動画、GitHubのIssuesやDiscussions、サポートコミュニティ——これらすべてが、ユーザーの質問に答えるための貴重なコンテキストになりえます。Connectionsはそれらを統合し、アシスタントが横断的に参照できるようにします。

#### 設定場所

GitBookエディタ → 対象ドキュメントサイトの「Settings」→「Connections」から設定できます。

#### 現在利用可能なコネクション種別

* **GitHub Issues**（リポジトリ単位で追加）
* **GitHub Discussions**
* **YouTube**（チャンネルIDを入力）
* **Website**（URLを指定。パス指定で特定セクションのみ対象にすることも可能）

追加希望のコネクション種別は `github.com/GitbookIO` のDiscussionsでリクエストできます。

#### 設定の詳細

Webサイトコネクションを例に、設定のポイントを説明します。

**URL指定の粒度** `synclyapp.com/blog` のように特定パス以下のみを対象にすることも、`synclyapp.com` のようにサイト全体を対象にすることも選択できます。ブログ記事だけを参照させたい、特定のサポートページだけを使いたいといったケースに対応できます。

**Search Ranking Boost（検索ランキングブースト）** アシスタントが回答生成時に、このコネクションをどの程度優先するかを数値で設定できます。公式ドキュメントをメインソースとして重視し、コネクションは補助的に使いたい場合は低めに設定し、特定のコネクションを積極的に使わせたい場合は高く設定します（デモでは1.2を使用）。

#### 動作の確認

コネクションを追加すると、GitBookはクロールを開始し、検出されたページ数・URLがリスト表示されます。クロール完了後は、アシスタントの検索時にコネクション由来のコンテンツが候補として表示されるようになります。

デモでは、「boosting productivity（生産性向上）」というキーワードで検索したところ、コネクション登録直後のブログ記事「Why Calendars Are Important」と「5 Ways Calendars Boost Your Productivity」がアシスタントの回答ソースとして表示されました。

さらに、AI Insightsパネルの個別質問詳細ビューにある「Sources（ソース）」欄でも、コネクション由来のソースがどの質問に使われたかが確認できるようになります。コネクションが実際に機能しているかどうかを追跡できる仕組みです。

#### YouTubeコネクションについて補足

参加者からの質問に対し、Addison氏は「YouTubeコネクションは動画のdescription（説明文）を使用しており、transcriptについても利用している可能性がある（要検証）」と回答しました。

### 5. Embeddable Assistant（埋め込みアシスタント）

#### 概要

埋め込みアシスタントは、GitBook Assistantをドキュメントサイトの外——自社Webサイト、プロダクト画面、LP等——に埋め込む機能です。ドキュメントサイトに登録されたすべての知識（コネクションを含む）を保持したまま、任意の場所でアシスタントを動作させることができます。

以前からベータ版として存在していましたが、今回は大幅な機能強化と合わせて正式な位置づけで発表されました。

#### 実装方法（3種類対応）

| 方法             | 概要                                   |
| -------------- | ------------------------------------ |
| Scriptタグ       | 最も簡単。WebサイトのHTMLにスクリプトを1行追加するだけで動作する |
| NPM            | npm installでパッケージとして導入               |
| React Provider | ReactアプリにProvider形式で直接統合できる          |

さらに、\*\*AI Coding Agent向けのSkill（スキル）\*\*も新たに公開されました。Claude CodeやCursorなどのコーディングエージェントにこのスキルを追加すると、「このWebサイトにGitBook Assistantを実装してください」と指示するだけで、ほぼ自動的に実装が完了するとのことです。

#### 設定場所

GitBookエディタ → Settings → Docs Embed

#### カスタマイズ可能な項目

* アシスタントの表示位置（左・右等）
* Ask ボタンの表示/非表示（ボタンを非表示にして、独自のUIからアシスタントを起動することも可能）
* アシスタントの名称
* アイコン・スタイル（サイトのカスタマイズ設定から自動反映）
* 表示形式：フローティングパネル / iframeコンテナへの埋め込み / フルスクリーン

なお「Powered by GitBook」のクレジット表示は非表示にできない仕様です。

#### デモで実演された使い方

デモでは、Synclyアプリの模擬Webサイトに以下のような実装が示されました。

1. スクリプトタグを追加しただけで、ページ右下にAskボタンが出現
2. 入力フォームのテキストをGitBook Assistantに直接パイプ接続し、Webサイト上の入力フォームで質問するとアシスタントが回答する構成を実現
3. GitBook自身のWebサイトでは、「Calculate savings（削減効果を計算）」「Explore features（機能を調べる）」などのアクションボタンが実装されており、ユーザーがクリックすると質問フローが起動する

#### カスタムアクション（ツール）の定義

埋め込みアシスタントは単純なチャットボットとして動作させるだけでなく、カスタムツールを定義することで多様な動作を実現できます。

* ツールの「いつ使うか」を説明文で定義する
* `execute` ファンクション内に任意のコードを記述できる
* 例：「ユーザーが特定の質問をした際にメール送信を実行する」「フォームに情報を記入する」「プロダクトのAPIを呼び出す」

参加者からの質問に対し、Addison氏は「ユーザーが特定の質問をしたときにメール送信もできる」と確認しています。チャットボットという枠を超え、プロダクトと深く統合されたインタラクティブなAIインターフェースとして機能させることが可能です。

また、アシスタントはユーザー情報の収集用途にも使えます。「製品ドキュメントの主な目的は？」「どのワークフローを自動化したいか？」といった質問フローを設計し、その回答をもとにパーソナライズされた回答を返す、という実装例もデモで示されました。

#### AI Insightsとの連携

埋め込みアシスタント経由の会話も、AI Insightsパネルにすべて記録されます。インサイトパネルの「Where（どこから）」欄で、質問がドキュメントサイト上から来たのか、埋め込みアシスタントから来たのかをソース別に確認できます。

#### 認証への対応

ドキュメントサイトが認証（Auth）で保護されている場合、埋め込みアシスタントも同様に認証済みリクエストのみ処理するため、セキュリティは維持されます。

### 6. Channels（チャンネル）※早期アクセス段階

#### 概要

Channelsは、GitBook Agentの機能をSlack・GitHubなど外部のコミュニケーションツール内から呼び出せるようにする仕組みです。現在は早期アクセス（ベータ）段階であり、すべてのユーザーがアクセスできるわけではありません。セミナーでは「現在積極的に開発中」と繰り返し強調されました。

#### チャンネルの機能（2軸）

**① アシスタント質問機能** SlackやGitHub内でGitBookボットにメンションし、「このドキュメントにはこの機能が説明されているか？」「現在の設定ではこの操作はどう動くか？」といった質問を、ドキュメントを参照した上で即座に回答させることができます。

**② エージェント機能（Change Request自動生成）** GitBook Agentが変更リクエスト（Change Request）を自動生成し、チームがGitBook上でレビュー・マージできる状態にします。このAgentはGitBook内にもともと組み込まれており、ドキュメントの編集・更新・既存Change Requestのレビューを行うことができます。Channelsはそのパワーを外部ツールから呼び出せるようにするものです。

#### GitHubチャンネルのデモ

デモでは、SynclyアプリのGitHubリポジトリにGitBookボットをCollaboratorとして追加した状態が実演されました。

1. PRを作成（架空のAPIエンドポイントを追加するというシナリオ）
2. PRのコメント欄に「このPRをレビューして、GitBookにドキュメント化してください」と投稿
3. GitBook Agentが変更内容を読み込み、既存のドキュメント構造を検索
4. 「既存にAPIページが見当たらないので、構造を確認しながら実装する」とコメントで返信
5. GitBook側にChange Requestが自動生成される

デモ中はバグにより最終的なChange Requestの表示に問題が生じましたが（Addison氏は「開発中に誰かが破壊的な変更をプッシュしたかもしれない」とユーモラスに言及）、コンセプトと動作の流れは明確に実演されました。

#### Slackチャンネルの概念

Slackでも同様の仕組みが機能する予定です。たとえばサポートチームが「このフィーチャーはまだドキュメント化されていない」と気づいた際に、GitBookボットをSlackでメンションするだけで、Change Requestが自動生成されます。

#### 将来ビジョン

Channelsの最終的なゴールは、ドキュメントの「自律的な自己更新」の実現です。

AI Insightsで検出された未回答・低品質な質問に対して、Agentが自動的に関連ページを更新・補強するChange Requestを提案し、チームがレビューしてマージするだけで、ドキュメントが常に最新状態に保たれる——という仕組みを構築することが目指されています。

人間のレビューを挟みながら、AIがドキュメントを継続的に改善していくというアプローチです。

### 7. Q\&Aセッション 主要質疑応答

セミナーの最後にQ\&Aが行われました。参加者からの質問と、Addison氏の回答をまとめます。

**Q：Estimated Savingsの計算方法をカスタマイズできるか？** A：現時点では不可。GitBook内部の指標を基に算出している。有用なフィードバックとして検討中。

**Q：オンプレミス環境でAIアシスタントは使えるか？** A：GitBookはオンプレミスインストールを提供していないため、完全なオンプレミス環境では動作が難しい。

**Q：AI機能の価格はどうなるか？** A：多くのAI機能をベータ期間中として提供してきたが、採用が増えるにつれてコストが増加するため、価格設定を積極的に検討中。公正で透明な価格体系を目指しているが、具体的な内容はまだ未定。

**Q：GitHubリポジトリ自体（コードベース）をコネクションとして追加できるか？** A：GitHub IssuesおよびGitHub Discussionsはリポジトリ単位でコネクションとして追加可能。コードそのものについては現時点では別途確認が必要。

**Q：YouTubeのトランスクリプトはコネクションに活用されるか？** A：動画のdescription（説明文）は確実に使用。transcriptについても利用している可能性があるが、要検証。実際にAddison氏自身が自分の動画で試したところ、動画内でのみ言及した内容をアシスタントが拾えたとのこと。

**Q：アシスタントは多言語対応しているか？** A：対応している。異なる言語で質問すれば、その言語で自動的に回答する。AI Insightsパネルでも質問者の言語情報が確認できる。

**Q：アシスタントのカスタマイズ範囲は？** A：アイコン・名称・スタイル・カスタムアクション等を広くカスタマイズできる。「Powered by GitBook」のクレジット非表示は不可。将来的にはチャットウィンドウ形式以外の表示形式も検討中。

**Q：カスタムツールでメール送信等は実装できるか？** A：可能。ツールの`execute`ファンクション内に任意のコードを記述でき、「ユーザーが特定の質問をした際にメール送信」のような処理も実装できる。

### 8. 所感・まとめ

今回のセミナーで提示されたGitBookの戦略を一言で表すなら、「ドキュメントを静的なコストセンターから、自己改善する動的資産へ」という転換です。

AI閲覧比率40%超という数字は、ドキュメントがAIサプライチェーンの重要な構成要素になったことを示しています。ドキュメントの質が、AIを通じて製品の価値に直接影響する時代になっています。

各機能がフィードバックループの中でどのような役割を持つかを整理すると以下のようになります。

<table><thead><tr><th width="211.63671875">機能</th><th>ループ上の役割</th></tr></thead><tbody><tr><td>AI Insights</td><td>何が足りないか（ギャップ）を可視化する → ドキュメント改善の起点</td></tr><tr><td>Connections</td><td>分散した社内外のナレッジを統合し、アシスタントの回答品質を向上させる</td></tr><tr><td>Embeddable Assistant</td><td>あらゆる接点でアシスタントを提供し、質問データの収集範囲を拡大する</td></tr><tr><td>Channels</td><td>開発現場（Slack・GitHub）からドキュメントを自動更新する → ループを閉じる</td></tr></tbody></table>

製品開発の観点から特に注目したいのは、**埋め込みアシスタントのカスタムツール機能**と、**チャンネルによる自動ドキュメント更新**の組み合わせです。Claude CodeやCursorなどのコーディングエージェント向けにSkillが提供されているという点も、開発ワークフローとドキュメント管理を統合するアプローチとして実用的です。

「ドキュメントを書くこと」から「ドキュメントが自律的に最新状態を維持するシステムを設計すること」へという方向性は、翻訳・多言語発信を含むコンテンツ管理全般に示唆を持つものだと感じます。
